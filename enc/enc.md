为了构建一个“工业级”且“全能型”的 Go 加解密封装库，我们需要将 **标准库 (`crypto/*`)** 的稳健性与 **扩展库 (`golang.org/x/crypto/*`)** 的先进性结合起来。

以下是为您整理的完整、系统化列表，分为**对称加密**、**非对称加密**、**哈希与消息认证**、以及**密钥派生**四个维度。

---

### 一、 对称加密 (Symmetric Encryption)
这是封装库的核心，必须同时支持“现代 AEAD 模式”和“经典 Block 模式”。

| 算法分类 | 具体算法/模式 | 所在包 | 封装建议与状态 |
| :--- | :--- | :--- | :--- |
| **现代认证加密 (AEAD)** | **AES-GCM** | `crypto/aes` | **首选**。高安全性，自带完整性校验，硬件加速快。 |
| | **ChaCha20-Poly1305** | `x/crypto/chacha20poly1305` | **推荐**。软件实现极快，适合移动端/无硬件加速设备。 |
| **经典块加密 (CBC)** | **AES-CBC** | `crypto/aes` | **必备**。对接 Java/C# 最常用。**必须手动实现 PKCS7 填充**。 |
| | **DES / 3DES-CBC** | `crypto/des` | **兼容**。仅用于银行、电力等老旧遗留系统，安全性低。 |
| **流加密 (Stream)** | **AES-CTR** | `crypto/aes` | 用于特定协议（如 SSH），不带校验，需谨慎使用。 |
| | **ChaCha20** | `x/crypto/chacha20` | 纯流加密，不带 Poly1305 认证。 |
| **特殊用途** | **AES-XTS** | `x/crypto/xts` | 专门用于**磁盘分区/扇区级**加密。 |

---

### 二、 非对称加密与签名 (Asymmetric)
封装时应涵盖从经典的 RSA 到现代的曲线算法。

| 算法分类 | 具体算法 | 所在包 | 封装建议与状态 |
| :--- | :--- | :--- | :--- |
| **经典加解密/签名** | **RSA (PKCS#1 / OAEP)** | `crypto/rsa` | **工业标准**。需支持 2048/4096 位，支持 PEM 格式转换。 |
| **常用签名** | **ECDSA (P256/P384)** | `crypto/ecdsa` | **主流**。用于 TLS 证书、区块链签名。 |
| **现代签名** | **Ed25519** | `crypto/ed25519` | **推荐**。性能极高，抗误用性强，API 简单。 |
| **密钥交换 (DH)** | **X25519 (Curve25519)** | `x/crypto/curve25519` | 用于生成临时会话密钥，TLS 1.3 核心。 |
| **高层封装** | **NaCl Box** | `x/crypto/nacl/box` | 组合算法，适合点对点安全通信，自动处理 Nonce。 |

---

### 三、 哈希与消息认证 (Hash & MAC)
用于数据完整性校验和 API 签名。

| 功能分类 | 具体算法 | 所在包 | 封装建议与状态 |
| :--- | :--- | :--- | :--- |
| **标准哈希** | **SHA-256 / SHA-512** | `crypto/sha256` | **必备**。目前最安全通用的摘要算法。 |
| | **SHA-3 (Keccak)** | `x/crypto/sha3` | 下一代标准，抵抗已知的 SHA-2 潜在攻击。 |
| **现代哈希** | **BLAKE2b / BLAKE2s** | `x/crypto/blake2b` | 比 SHA-3 更快，安全性极高。 |
| **消息认证** | **HMAC (with SHA256)** | `crypto/hmac` | **必备**。用于 API 请求签名，防止中途篡改。 |
| | **Poly1305** | `x/crypto/poly1305` | 高性能 MAC，常与 ChaCha20 配合。 |
| **兼容/不安全** | **MD5 / SHA-1** | `crypto/md5` | **仅限校验文件**。严禁用于加密或安全认证。 |

---

### 四、 密钥派生与密码存储 (KDF)
这是处理用户登录密码、从弱密码生成强密钥的关键部分。

| 功能分类 | 具体算法 | 所在包 | 封装建议与状态 |
| :--- | :--- | :--- | :--- |
| **用户密码哈希** | **Bcrypt** | `x/crypto/bcrypt` | **Go 社区标准**。自带盐值，抗暴力破解。 |
| | **Argon2 (id/i)** | `x/crypto/argon2` | **目前最强**。抗 ASIC/GPU 破解，适合高安全需求。 |
| **密钥衍生** | **HKDF** | `x/crypto/hkdf` | 将一个种子密钥派生出多个不同用途的密钥。 |
| | **Scrypt** | `x/crypto/scrypt` | 用于从密码生成高强度的对称密钥。 |
| | **PBKDF2** | `x/crypto/pbkdf2` | **经典**。兼容旧版系统或特定标准（如 WPA2）。 |

### 🛠️ 封装时的核心“坑点”提醒（避坑指南）

1.  **关于 AES-CBC 的填充：**
    Go 标准库默认不提供填充。封装时必须实现 `PKCS7Padding`，否则你加密的数据 Java 无法解开，Java 加密的数据你也解不开。
2.  **关于 IV (初始化向量)：**
    永远不要让业务层去传 IV。你应该在 `Encrypt` 内部随机生成 IV，并将其放在密文的最前面；在 `Decrypt` 时提取前 16 字节（AES 块大小）作为 IV。
3.  **关于 `crypto/rand`：**
    所有涉及生成随机 Key、Salt、Nonce、IV 的地方，必须使用 `crypto/rand`，绝对不能用 `math/rand`。
4.  **关于 RSA 格式：**
    封装时要提供 `ParsePrivateKeyFromPEM` 和 `ParsePublicKeyFromPEM`，因为不同平台给你的可能是 PKCS1（`BEGIN RSA PRIVATE KEY`）或 PKCS8（`BEGIN PRIVATE KEY`）。
